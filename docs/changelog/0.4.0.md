# v0.4.0 — Achievement Scheduling Redesign & Game Queue

## EN

### Breaking Changes

- **Game status renamed**: `Running` → `Playing`
- **DB schema changes**: `game.target_hours`, `game.total_play_minutes`, `game.last_session_start`, `achievement.unlock_offset_minutes` removed; new fields and tables added (see below)
- **Existing scheduled offsets invalidated**: upgrade requires re-Refresh of all in-progress games

### Architecture

- **Achievement scheduling redesign** — replaced offset-based scheduling (`unlock_offset_minutes` + per-game timer) with absolute-timestamp scheduling (`scheduled_unlock_at`). At Refresh, each achievement is assigned a concrete UTC unlock time derived from a linked reference player's actual unlock intervals (Method A: original relative offsets preserved). A global scanner fires every 60 s and unlocks any achievement whose `scheduled_unlock_at ≤ NOW()`.
- **Playtime tracking redesign** — removed internal play-minute counter. Playtime is now tracked via Steam API re-anchoring: at each Start/Refresh, current Steam playtime is snapshotted (`steam_playtime_at_refresh`). MCT reached when elapsed session time ≥ `target_minutes − steam_playtime_at_refresh`.
- **New game states** — `Idle → Playing → Scheduled → Completed`. `Scheduled` = MCT reached, game offline, achievements still unlocking per schedule. `Completed` = MCT reached AND all achievements unlocked.
- **MCT as target playtime** — game target playtime now sourced from SteamHunters Median Completion Time (MCT) rather than a user-configured `target_hours`. Fallback: reference player's actual total unlock duration.
- **Server restart recovery redesign** — Playing games are re-anchored to server start time (Steam API re-read + `scheduled_unlock_at` recalculated from server start time), preventing burst unlocks on restart.

### New Features

- **Reference player** — each game can be linked to a SteamHunters player URL (`game_reference_player` table). Refresh uses that player's achievement unlock timestamps to calculate scheduled times. Includes burst detection: if ≥ 5 achievements share the same timestamp, Refresh is blocked with an error (unless `override_burst_check` is enabled on the player link).
- **Game queue** — dedicated idle queue (`game_queue` table) for games pending playtime. Only games with a reference player and a completed Refresh can be queued. Supports drag-and-drop reordering. Dashboard Start button fills up to 2 concurrent Playing slots from the queue.
- **Max 2 concurrent games** — at most 2 games may be in Playing state simultaneously; attempting to Refresh a 3rd errors out.
- **Auto-advance queue** — when a Playing game reaches MCT and transitions to Scheduled, the queue auto-advances and starts the next queued game.
- **Force Refresh** — resets all achievements to locked, then re-runs the full Refresh flow.
- **ScheduledUnlockAt recalculation on queue start** — when a game is started from the queue, `scheduled_unlock_at` is recalculated from the actual start time, preventing stale schedules from a long wait in queue.

### UI Changes

- Dashboard: queue control panel with drag-and-drop reordering, Start/Stop buttons; Start disabled when all slots full; Stop disabled when nothing playing; actions now reload game list and stats immediately
- GameDetail: reference player config section (URL input + burst override toggle); achievement table updated (`Scheduled At` column, `Scheduled` status chip); loading overlay on Save; achievement sort — pending ASC by scheduled time, unlocked DESC by unlock time; time columns display `yyyy-MM-dd HH:mm`
- Games page: Refresh queues the game instead of starting play immediately; status chip gains `Scheduled`; queued games show yellow "Queued" chip (top-right); Add to Queue / Remove from Queue icon buttons (`QueuePlayNext` / `RemoveFromQueue`); button hidden when no reference player configured; thumbnail fallback for games without Steam header image (dark gradient + game name placeholder)
- Completed games: no play/re-queue action; filtered from queue list; status shown in top-right chip only

### Bug Fixes

- **Restart re-anchoring** — anchor offset now uses reference interval math: `anchor = now + max(0, refInterval − elapsed)`, preventing burst unlocks on service restart
- **DbContext concurrency** — `GameRefreshService` switched to `IServiceScopeFactory`; eliminates "A second operation was started on this context" error when Add to Queue fires during a Blazor re-render
- **Queue drag-and-drop** — items now rendered via `ItemRenderer` template instead of `@foreach`; drag-and-drop functional

### New Features

- **Configurable concurrent slots** — `MaxConcurrentGames` in `steam_config` (default 1); configurable in Settings → Queue Settings; enforced by Start Queue, auto-advance, and Dashboard button guards
- **Sync button on game cards** — Steam-only data pull (no SteamHunters); auto-marks Completed when playtime goal + all achievements met
- **Immediate Dashboard feedback** — Start Queue / Stop Queue / Remove from Queue reload game list and recompute stats without waiting for the 1-minute timer

## 中文

### 破坏性变更

- **游戏状态重命名**：`Running` → `Playing`
- **数据库 schema 变更**：`game.target_hours`、`game.total_play_minutes`、`game.last_session_start`、`achievement.unlock_offset_minutes` 已移除；新增字段与表（见下方）
- **已有成就解锁偏移量失效**：升级后需对所有进行中游戏重新执行 Refresh

### 架构变更

- **成就调度系统重构** — 将基于偏移量的调度（`unlock_offset_minutes` + 游戏内置计时器）替换为基于绝对时间戳的调度（`scheduled_unlock_at`）。每次 Refresh 时，以参考玩家的真实解锁间隔为基准，为每个成就计算具体 UTC 解锁时刻（方案 A：保留原始相对偏移）。全局扫描器每 60 秒运行一次，解锁所有 `scheduled_unlock_at ≤ 当前时间` 的成就。
- **游戏时长追踪重构** — 移除内部分钟计数器，改为基于 Steam API 重新锚定：每次 Start/Refresh 时读取当前 Steam 时长快照（`steam_playtime_at_refresh`）。达标条件：本次 session 已用时 ≥ `target_minutes − steam_playtime_at_refresh`。
- **新增游戏状态** — `Idle → Playing → Scheduled → Completed`。`Scheduled`：MCT 已达标、游戏已下线，仍有成就待按计划解锁。`Completed`：MCT 达标且全成就已解锁。
- **MCT 作为目标时长** — 游戏目标时长现在来源于 SteamHunters 中位完成时长（MCT），不再由用户手动设置 `target_hours`。Fallback：参考玩家实际总解锁时长。
- **服务重启恢复重构** — Playing 状态游戏重启后以服务启动时刻重新锚定（重读 Steam API + 重算 `scheduled_unlock_at`），避免重启后瞬间批量解锁成就。

### 新功能

- **参考玩家** — 每款游戏可关联一个 SteamHunters 玩家主页 URL（`game_reference_player` 表）。Refresh 时使用该玩家成就解锁时间戳计算计划时间。内置数据质量检测：同一时刻 ≥ 5 个成就同时解锁时报错中止（可在玩家链接中开启 `override_burst_check` 忽略）。
- **游戏队列** — 新增专用挂机队列（`game_queue` 表），仅配置了参考玩家且完成 Refresh 的游戏可入队。支持拖拽排序。Dashboard Start 按钮从队列中启动最多 2 个并发游戏。
- **最多 2 款游戏同时挂机** — Playing 状态上限 2 款，Refresh 第 3 款时报错。
- **队列自动推进** — Playing 游戏 MCT 达标转入 Scheduled 后，自动启动队列下一款游戏。
- **Force Refresh** — 重置所有成就为未解锁，再执行完整 Refresh 流程。
- **队列启动时重算 ScheduledUnlockAt** — 游戏从队列启动时以实际启动时刻重算所有成就计划时间，避免在队列中等待过久导致计划时间全部过期。

### UI 变更

- Dashboard：新增队列控制区（拖拽排序 + Start/Stop 按钮）；槽满时 Start 置灰，无游戏运行时 Stop 置灰；操作后立即刷新游戏列表和统计数据
- GameDetail：新增参考玩家配置区（URL 输入框 + 数据质量覆盖开关）；成就表格新增 `Scheduled At` 列和 `Scheduled` 状态 Chip；Save 按钮加全屏 Loading 遮罩；成就排序 — 待解锁按计划时间正序，已解锁按解锁时间倒序；时间列格式 `yyyy-MM-dd HH:mm`
- Games 页：Refresh 触发入队而非立即挂机；状态 Chip 新增 `Scheduled`；队列中游戏右上角显示黄色 "Queued" Chip；Add to Queue / Remove from Queue 使用专用图标按钮；未配置参考玩家的游戏不显示入队按钮；无 Steam 缩略图的游戏显示深色渐变 + 游戏名称占位符
- Completed 游戏：不可再Play/入队；从队列列表中隐藏；状态仅在右上角 Chip 显示

### 修复

- **重启重锚定** — 锚定偏移改用参考间隔数学计算：`anchor = now + max(0, refInterval − elapsed)`，避免重启后瞬间批量解锁
- **DbContext 并发错误** — `GameRefreshService` 改用 `IServiceScopeFactory`，解决 Add to Queue 与 Blazor 重渲染同时发生时的"second operation"异常
- **队列拖拽排序** — 改用 `ItemRenderer` 模板渲染队列项，拖拽功能恢复正常

### 新功能

- **可配置并发槽位数** — `steam_config` 新增 `MaxConcurrentGames`（默认 1）；可在 Settings → Queue Settings 中修改；Start Queue、自动推进和 Dashboard 按钮均遵守此配置
- **游戏卡片 Sync 按钮** — 仅从 Steam 拉取最新数据（不调用 SteamHunters）；满足时长+全成就条件时自动标记为 Completed
- **Dashboard 即时响应** — Start Queue / Stop Queue / Remove from Queue 操作后立即重载游戏列表和统计，无需等待 1 分钟定时器
